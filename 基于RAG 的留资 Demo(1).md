第一,**用状态机 + 槽位(slot)管路径,而不是让大模型自由发挥**。每个节点是一个明确的状态,有自己的提问、可点击的快捷按钮、要填的槽位(当前学历、提升目标、用途),以及"满足什么条件去下一个节点"的规则。按钮主导对话,把用户的选择限制在你画好的轨道里,这是"不开放"的第一道保险。

第二,**所有自由文本统一过一个意图识别,只分三类**。用户不点按钮、自己打字时,先分类再处理:属于知识库范围的事实问题 → 调 RAG 回答,答完用一句话把人拉回当前节点;表达了购买/资格/报名意图(比如"多少钱""我能不能报""怎么交费")→ 直接跳到留资钩子;跟业务无关 → 礼貌挡回("这个我帮不上,不过关于学历提升…")。这样无论用户怎么问,对话永远收敛回主线。

第三,也是最关键的:**把"钩子"写进 RAG 的生成规则里,让它故意不答透**。这是这类系统和普通客服机器人的本质区别。RAG 答一般性问题(几种学历形式的区别、学信网能不能查、政策科普)用来建立信任、打消顾虑;但凡涉及精确报价、个人能不能报、最快多久毕业、具体怎么报名这些,生成时一律收口到"这个要根据您的具体情况核算,我帮您约老师免费做一份规划"。**信息差是留电话的动力**——答得太透,用户就没有留资的理由了。

### 留资钩子的触发条件

钩子节点不是随便弹,而是在这几种时机触发转化率最高:用户问到被"收口"的问题(价格/资格/报名方式);连续答疑 N 轮、明显有兴趣之后;用户主动表达强意图("我想报""怎么报名");或用户直接要求联系老师。钩子话术的要点是**价值前置 + 低承诺**:不是"留个电话",而是"您这个情况符合 XX 政策,老师可以免费帮您算具体费用和最快拿证时间,方便留个手机号吗?只是初步沟通,不强制报名。"

### RAG 知识库怎么搭

文档按问题场景分类:学历形式科普(成考/自考/国开对比)、报考条件、费用区间(只给范围,精确报价留给销售)、政策与证书效力(学信网、考公考编评职称落户能不能用)、常见异议处理话术、院校专业库。

切分上,**FAQ 用「问—答」成对存储**,检索命中率最高;政策类长文按小节切。检索建议向量 + 关键词(BM25)混合再加一层重排(rerank),中文场景比纯向量准很多。生成层的约束 prompt 是重点:只用检索到的内容、不编造、语气亲切、每次结尾都带一句引导、遇到价格/资格类问题就转留资——这套规则就是上面说的"钩子"的落地。
